# Notes on Running gigaGPT on MacBook Pro 2014 Linux

## OS
Ubuntu 24.04.3 LTS

## Run configs/cpu_small.yaml
```shell
uv run python train.py configs/cpu_small.yaml
# CC decided to run this first
GENERATING_CEREBRAS_PYTORCH_DOCS=1 uv run -p 3.12 python data/shakespeare_char/prepare.py
# CC detected missing dependencies, changed to run
GENERATING_CEREBRAS_PYTORCH_DOCS=1 uv run -p 3.12 python train.py configs/cpu_small.yaml
# Training completed successfully. 100 steps ran, loss dropped from ~10.86 to ~6.35,
# with checkpoints saved at steps 50 and 100 to out/cpu_small/.
# The missing deps (h5py, hdf5plugin, zstandard) were added along the way.
```

## Why do you need to set GENERATING_CEREBRAS_PYTORCH_DOCS=1 when train?
When GENERATING_CEREBRAS_PYTORCH_DOCS=1 is set, the library skips two imports:
  1. from .backends import backends — the hardware backend drivers (CSX, GPU, etc.)
  2. from .lib import cerebras_pytorch_lib — a compiled C extension .so from the wheel
  Both of these were compiled for Linux x86_64 and cannot load on macOS ARM64.
  The flag was originally meant for doc generation, but it happens to also bypass the platform-specific
  compiled code, which is exactly what's needed to run on macOS ARM64 where those .so files are absent.

  In short: without the flag, the import crashes because it tries to load Linux x86_64 native libraries. With it, only the pure-Python parts load — which is enough for CPU-backend
  training.
