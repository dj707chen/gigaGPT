# Notes on Running gigaGPT on MacBook Pro 2014 Linux

## OS
Ubuntu 24.04.3 LTS

## Run configs/cpu_small.yaml
```shell
uv run python train.py configs/cpu_small.yaml
# CC decided to run this first
GENERATING_CEREBRAS_PYTORCH_DOCS=1 uv run -p 3.12 python data/shakespeare_char/prepare.py
# CC detected missing dependencies, changed to run
GENERATING_CEREBRAS_PYTORCH_DOCS=1 uv run -p 3.12 python train.py configs/cpu_small.yaml
# Training completed successfully. 100 steps ran, loss dropped from ~10.86 to ~6.35,
# with checkpoints saved at steps 50 and 100 to out/cpu_small/.
# The missing deps (h5py, hdf5plugin, zstandard) were added along the way.
```

### Why do you need to set GENERATING_CEREBRAS_PYTORCH_DOCS=1 when train?
When GENERATING_CEREBRAS_PYTORCH_DOCS=1 is set, the library skips two imports:
  1. from .backends import backends — the hardware backend drivers (CSX, GPU, etc.)
  2. from .lib import cerebras_pytorch_lib — a compiled C extension .so from the wheel
  Both of these were compiled for Linux x86_64 and cannot load on macOS ARM64.
  The flag was originally meant for doc generation, but it happens to also bypass the platform-specific
  compiled code, which is exactly what's needed to run on macOS ARM64 where those .so files are absent.

  In short: without the flag, the import crashes because it tries to load Linux x86_64 native libraries. With it, only the pure-Python parts load — which is enough for CPU-backend
  training.

Cerebras SDK is at ~/Download/CerebrasSDK
Two blockers here:
 1. This SDK is the CSL (Cerebras Systems Language) SDK — it contains tools for low-level hardware programming (cslc compiler, csdb debugger, sdk_debug_shell). This is a different layer
  from the PyTorch-level simfab that train.py connects to via mgmt_address.
  2. Singularity/Apptainer is not installed — all tools in this SDK run inside the .sif Singularity container, so nothing here can run yet.
  To run train.py with backend: "CSX" you need two separate things:
  ┌──────────────────────────┬────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │           What           │                                                                   Where                                                                    │
  ├──────────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Singularity or Apptainer │ Install on this machine — sudo apt install apptainer                                                                                       │
  ├──────────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ PyTorch simfab server    │ A separate Cerebras appliance server (the cbcore cluster-mgmt service) that train.py connects to as mgmt_address. This is not in this SDK. │
  └──────────────────────────┴────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
  The PyTorch simfab is part of the Cerebras Appliance software stack (the server-side counterpart to cerebras-appliance PyPI package), typically deployed as a Kubernetes cluster or a
  dedicated VM image. It's distributed separately from the SDK — you'd need to request it from Cerebras.

  What you can do now: Install Apptainer if you want to use the CSL tools in this SDK:
  sudo apt install apptainer
  But that won't unblock train.py with CSX backend — that needs the appliance server.

Can Cerebras PyTorch simfab server run on Mac?
Yes, you can run the Cerebras PyTorch SDK on an Apple Silicon Mac, but it is typically done through a
Linux virtual machine environment. According to Cerebras [documentation](https://sdk.cerebras.net/installation-guide  ), the recommended setup for using
the Cerebras SDK on an Apple Silicon Mac is via a Lima Linux virtual machine.
- Supported Setup: Using Lima (Linux on Mac) allows for the necessary Linux-based environment.
- Performance: The documentation recommends using Rosetta within the virtual machine setup for improved performance.
- Purpose: This setup enables you to develop, compile, and test code aimed at the Cerebras Wafer-Scale Engine (CS-2/CS-3 systems) on a local Mac workstation.
- Simfab/Simulation: You can run simulations using the simulator=True option within the SDK on these machines to test code without a physical CS-2 system.
